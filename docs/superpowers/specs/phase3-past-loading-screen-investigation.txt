Investigation: what's actually stalling past the loading screen

Goal: find why the game never visibly progresses past the loading screen, now that
the file-I/O + APC + spinlock + sprintf chain (Phase 3H-3J) is fixed and the game
runs a real, continuous per-frame loop.

--- Finding 1: the "healthy 2 calls/sec" measurement was wrong ---

Initial call-rate instrumentation on sub_8212EC78 (the queue-poll wrapper fixed in
Phase 3H) mislabeled sparse activity as "~1s" cadence. Fixed to report real elapsed
time. Corrected result: sub_8212EC78 is called a total of TWO times, ever, in the
entire post-file-load execution -- not a steady rate. A 4-minute patience run (real
wall-clock, watchdog temporarily raised) confirmed zero further kernel-visible
activity for the remaining ~235 seconds, while CPU stayed pegged at 100%+. This is
a second, genuinely different stall from anything fixed in Phase 3H-3J -- the queue
system itself is not looping; something else entirely is.

--- Finding 2: real call-stack structure of the stuck state ---

Repeated lldb sampling of the active thread found a stable 11-deep call chain that
persists for minutes, with the leaf function varying (real, ongoing execution, not
a frozen PC):

  __xstart -> sub_825DEC18 -> sub_8212B130 -> sub_8244B040 -> sub_8244AFB0 ->
  sub_8244AE70 -> sub_82461D28 -> sub_825E8D28 -> sub_82446D50 -> sub_8212EC78 ->
  sub_8212EAF0 -> [varies: sub_8212EA80 / sub_82125DF8 / mutex lock]

Read each function's real generated code (private/ppc/*.cpp) to find the actual
loop. Confirmed NOT loops (all-forward-branch dispatchers or straight-line code):
  - sub_825DEC18: calls sub_8212B130 exactly once, then a permanent `b $` trap --
    this is the correct, expected "title entry point should never return" pattern,
    not a bug.
  - sub_8244B040: large (~1080 line) dispatcher, all branches forward, no loop.
  - sub_825E8D28: jump-table switch dispatcher, all branches forward, no loop.
  - sub_82446D50: one small, correctly-bounded input-state bit-packing loop
    (iterates a real count field, terminates normally), otherwise straight-line.

Confirmed REAL loops, both apparently correctly bounded:
  - sub_82461D28 (private/ppc/ppc_recomp.59.cpp:25072-25323): iterates a real
    item list, count reloaded from memory each iteration, dispatches each item via
    an indirect (vtable-style) function call. Live-instrumented: count was a sane
    3, and the loop-back check was hit only once in the whole run -- meaning
    item[0]'s own indirect call is what doesn't return, not the loop itself.
  - sub_8244AE70 (private/ppc/ppc_recomp.57.cpp:1801-1831): a tight retry loop --
    call sub_8212EC78(), call sub_82128EB0(fixedAddr), check the low 16 bits of
    the result, loop again if zero. This is the most likely site of the real
    stall: a classic "wait for condition via repeated polling" shape, the same
    class of bug found and fixed in Phase 3H.

--- Finding 3: sub_82128EB0 / sub_82128DB0 lead into real ambiguity ---

sub_82128EB0 (private/ppc/ppc_recomp.12.cpp:1590) branches on a global flag at
[-32158 base + 10104]:
  - flag == 0 (the branch taken on essentially every call, per live evidence):
    treats its first argument (a fixed address computed by the caller, not
    obviously a function pointer at the source level) as a function pointer and
    calls it directly via mtctr/bctrl, then unconditionally returns 1 (success).
  - flag != 0: calls sub_82128DB0 for real queue submission against the same
    0x82670000+5632 structure central to Phase 3H-3G's earlier investigation
    (checks field4 against a cap of 128, submits into a slot, decrements/increments
    bookkeeping, wakes via sub_82128D08 if field12 is set).

Live instrumentation at sub_82128DB0's field4 check found: field4=0 (queue not
full -- my initial "queue full forever" hypothesis was wrong), and this check
point was reached only ONCE in the entire run. This means the flag==0 branch (the
indirect bctrl call to a caller-supplied address) is the path actually being taken
on essentially every retry iteration -- and that indirect call is the most likely
real hang: either the target address doesn't point to code that returns quickly,
or (an open possibility, not confirmed) the address is genuinely data being
mis-treated as code, which would itself be a real bug matching the "call an
address as if it always contains valid trampoline code" pattern already seen once
this session in a different function.

--- Finding 4: the fixed-address hypothesis was traced and then refuted ---

Traced the exact constant sub_8244AE70 passes to sub_82128EB0: `-32187 base
(0x82450000) - 20936 (0x51C8) = 0x8244AE38`. Confirmed this address IS a real,
mapped function (`sub_8244AE38` appears in private/ppc/ppc_func_mapping.cpp) --
not an unmapped/null indirect-call target as initially suspected. Read it: short,
straight-line, calls sub_82495908 once, always returns 1. Read sub_82495908: a
larger function containing a real circular-linked-list traversal (walk via a
"next" pointer at offset 20 until back to a saved sentinel node) -- structurally
the same shape as the XDVDFS BST bug fixed earlier this project, a plausible
timer/callback-list walk.

Instrumented that list-traversal loop's iteration count live: zero hits in a
15-second run -- the loop is never entered at all.

Went further and used lldb breakpoints directly (attach to an already-running
process, breakpoint on `__imp__sub_8244AE38`, continue) rather than continuing to
reason from static code alone. Result across a full 60-second real run: the
breakpoint never triggers. This is a decisive, empirical negative result --
sub_8244AE38 is never actually called during real execution, which means the
flag-==0 branch of sub_82128EB0 (the bctrl-to-caller-address path) is NOT the one
being taken on the hot retry path, contradicting the single live sample that
suggested otherwise. My reconstructed call chain from sub_8244AE70 down through
this specific branch does not reflect what actually happens at runtime.

--- Finding 5: symbol breakpoints are unreliable under this build's -O3, and the
    "retry loop" hypothesis for sub_8244AE70 is fully refuted ---

Tried the planned next step (breakpoints on `__imp__sub_82128DB0` and
`__imp__sub_8212EC78` directly) and got zero hits across a full 60-second real
run on both -- despite independently confirmed evidence (kernel log output,
earlier source-level instrumentation) that these functions DO execute. Root
cause: this project builds Release / -O3 (confirmed in build/CMakeCache.txt).
Small, hot, directly-called (`bl`-style) functions like these are prime inlining
candidates, and an inlined function has no distinct call boundary left for a
symbol breakpoint to catch. This does NOT invalidate the earlier sub_8244AE38
negative result, though -- that address is only ever reached via an indirect
`bctrl` (PPC_CALL_INDIRECT_FUNC), and indirect/computed call targets cannot be
inlined by the compiler, so that breakpoint result stands.

Given breakpoints are unreliable here, went back to the technique proven
reliable all session: instrumenting the function body directly (inlining-proof,
since the print statement travels with the code wherever it ends up). Added a
watch at the exact point inside sub_82128EB0 that reads the branch-selecting
flag and (for the flag==0 case) the bctrl target, printing both on every real
entry to the function.

Result (full run, real evidence): sub_82128EB0 is entered exactly ONCE, ever,
with flag=1 (not 0) -- meaning it takes the sub_82128DB0 branch, not the bctrl
branch, on its one and only real call. This directly means the "loc_8244AF60"
loop inside sub_8244AE70 (call sub_8212EC78, call sub_82128EB0, loop if the
result's low 16 bits are zero) is NOT a tight retry loop stuck spinning -- it
runs through once and, since sub_82128DB0's real submission succeeded (field4=0,
not full, per earlier evidence), should return a non-zero result and exit
normally. The entire "sub_8244AE70 is the hang site" hypothesis, built from
several real, confirmed sub-findings, is refuted by this final, decisive,
inlining-proof measurement. Whatever the real 4-minute stall is, it is not this
loop.

This pass ends with several confirmed real facts and one large invalidated

--- Finding 6: ROOT MECHANISM FOUND -- sub_8212EC78's own inner loop is the
    real hang, and its exact stuck state is now confirmed with hard numbers ---

Instrumented sub_8212EAF0's call rate directly (elapsed-time-corrected, not the
earlier flawed labeling): confirmed ~9.6 MILLION calls per second, sustained for
the entire multi-minute hang. This is unambiguous: a genuine tight busy-spin,
not slow legitimate work.

Traced the caller: this rate exactly matches sub_8212EC78's OWN internal loop
(read fully back in an earlier phase) -- `call sub_82128F58(); if it returns 0,
loop: call sub_8212EAF0(8); check field12; if field12!=0, call sub_82128F58()
again; loop again if it returns 0`. Instrumented that inner loop directly:
confirmed `field4=1` and `field12=1`, STUCK at those exact values, unchanging,
for the entire run (verified samples at hit=1 through hit=20,000,000+).
sub_82128F58() keeps returning 0 (not-clear) every single time. This is the
literal, complete mechanism of the hang: sub_8212EC78 never returns because the
queue never reports clear, so it spins its own internal poll loop forever,
burning the CPU via sub_8212EAF0/sub_82128F58 millions of times a second. This
directly explains every earlier observation in this investigation: sub_8212EC78's
own OUTER call count stays low (2 total, matching Finding 1) because it is
STUCK inside one of those two calls, never returning to be called again; the
9.6M/s sub_8212EAF0 rate; the 100%+ pegged CPU; the complete absence of further
kernel-visible activity (nothing else runs because this thread never leaves
the loop).

This is the exact same field4/field12 mechanism investigated and fixed for the
loading-screen texture in Phase 3H -- but a DIFFERENT, later occurrence of it.
Instrumented NtQueueApcThread and TryDeliverOneApc (the real APC machinery from
Phase 3H) directly to log every real call: confirmed exactly ONE
NtQueueApcThread + ONE TryDeliverOneApc pair happens in the entire run, and it
is the SAME loading-screen APC already fixed (target=0x1005,
routine=0x820A98C0, context=0x8212CAC8 -- matches the Phase 3H commit's
documented values exactly). That APC delivers successfully. Immediately after
it runs, field4 is stuck at 1 -- meaning the APC's own routine (0x820A98C0),
when it executes, submits a NEW item into the same queue as part of its own
logic (plausibly "resource loaded, now do X with it"), and NOTHING ever queues
a second APC (or any other completion signal) to clear THIS submission. The
consumer side of Phase 3H's fix works correctly; there is a second, distinct
producer-side gap this time.

--- Finding 7: traced routine 0x820A98C0 to the real submission call site, and
    found (then reverted) an unsafe fix built on a wrong hypothesis ---

0x820A98C0 itself is a thin, generic trampoline (`r11=r3; r3=r4; mtctr r11;
bctr`) -- it just jumps to whatever address is in its own first argument,
passing its second argument along. Given NtQueueApcThread's captured params
(routine=0x820A98C0, context=0x8212CAC8, sysArg1=0x0) and
TryDeliverOneApc's real calling convention (r3=apc.context, r4=apc.sysArg1),
the REAL callback this trampoline jumps to is 0x8212CAC8, called with arg=0.

Read sub_8212CAC8 (450 lines, real scene/layer processing logic -- 3-way
cycling via div-by-3, per-object dispatch). Bracketed its execution with a
field4/field12 watch: confirmed precisely `field4=0` on entry (clear) and
`field4=1` on exit (stuck) -- this is unambiguously the function that performs
the real submission. It does not call the known submission helpers
(sub_8212B210/sub_82128DB0) directly, so the increment must happen inside a
callee. Found the real culprit by listing every call it makes and reading the
plausible candidates: sub_8212F788 directly calls `__imp__NtReadFile` -- a
SECOND read of the same already-open loading-screen file (handle 0x101C),
issued with an `ori r5,r5,1` bit-tag pattern suggesting the game is using
NtReadFile's Event/ApcRoutine/ApcContext parameters (r4/r5/r6 in the real NT
signature) for real asynchronous completion -- parameters this project's
current NtReadFile implementation completely ignores (it only reads
r7-r10: IoStatusBlock/Buffer/Length/ByteOffset).

Instrumented NtReadFile directly to log the previously-ignored r4/r5/r6.
Confirmed live: `event(r4)=0x0`, `apcRoutine(r5)=0x8212F721`,
`apcContext(r6)=0x8212A9E8` -- non-null values on exactly this second read,
matching the theory that this is real async I/O with a real completion
callback the game expects and our implementation currently drops.

Implemented a fix: when apcRoutine != 0, queue a real APC (reusing the exact
Phase 3H machinery) to the calling thread with routine=apcRoutine,
context=apcContext, sysArg1=ioStatusBlockPtr -- matching real IoApcRoutine's
signature (ApcContext, IoStatusBlock, Reserved), which lines up exactly with
TryDeliverOneApc's existing (r3=context, r4=sysArg1, r5=0) calling convention.

Testing this fix caused a real crash (EXC_BAD_ACCESS, captured via lldb
attach at address 0x10259 -- clearly not a valid PPC code address). Checked
whether 0x8212F721 (the captured "apcRoutine" value) is actually a real,
mapped function: it is NOT in private/ppc/ppc_func_mapping.cpp, and critically
it is ODD (ends in 0x21, not 4-byte aligned) -- real PPC instructions and
therefore all real code addresses are always 4-byte aligned, so this value
cannot be a genuine function pointer as-is. The "IO completion APC" hypothesis
for r5/r6 is likely wrong, or Xbox 360's real NtReadFile signature differs
from desktop NT's in a way not yet understood, or r5's low bit is a real,
separate flag that needs masking before use rather than being part of a raw
address. The fix was reverted in full (host/kernel_impl.cpp and the temporary
watch in private/ppc/ppc_recomp.12.cpp) rather than shipped broken -- verified
clean via git status/diff and a rebuild.
