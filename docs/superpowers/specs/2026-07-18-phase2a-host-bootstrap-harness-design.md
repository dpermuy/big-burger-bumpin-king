# Phase 2A: Host Bootstrap Harness — Big Bumpin' Recomp

## Context

Phase 1 ([spec](2026-07-18-phase1-static-recompilation-pipeline-design.md)) produced
`libBigBumpinPPC.a`: the game's PowerPC code recompiled to C++ and compiling clean, but
not linked into anything runnable and with no runtime behind it.

Phase 2 of the overall roadmap is "minimal bootable runtime." That phase is itself too
large for one spec — the generated code references 210 unique unresolved Xbox 360 kernel
imports (`Ke*`, `Ex*`, `Hal*`, `Io*`, `XMA*`, `XAudio*`, etc. — counted directly from
`PPC_EXTERN_FUNC(__imp__...)` declarations in `private/ppc/*.h`). It splits into:

- **2A (this spec):** a host executable that allocates the guest memory image, wires up
  indirect function calls, and invokes the game's entry point. Every kernel import gets
  a mechanical placeholder stub so the executable actually links and runs.
- **2B (future spec):** replace stubs that get hit and misbehave with real implementations,
  driven by what 2A's run actually exercises — not a blind 210-item checklist.

## Facts From the Generated Code (verified, not assumed)

- `private/ppc/ppc_config.h`: `PPC_IMAGE_BASE = 0x82000000`, `PPC_IMAGE_SIZE = 0x740000`,
  `PPC_CODE_BASE = 0x820A0000`, `PPC_CODE_SIZE = 0x57A3B8`.
- `thirdparty/XenonRecomp/XenonUtils/ppc_context.h`: `PPC_MEMORY_SIZE = 0x100000000`
  (4 GiB) — the intended size of the `base` buffer passed to every `PPC_FUNC`.
- Indirect calls resolve via `PPC_LOOKUP_FUNC(base, addr)`, which reads a host function
  pointer from `base + PPC_IMAGE_BASE + PPC_IMAGE_SIZE + (addr - PPC_CODE_BASE) * 2`. The
  host harness must populate this table at startup.
- `private/ppc/ppc_func_mapping.cpp` defines `PPCFuncMapping PPCFuncMappings[]` — 28,489
  `{guest_address, host_function_pointer}` pairs, already compiled into `BigBumpinPPC`.
- The entry point is a real, directly callable C++ symbol: `_xstart(PPCContext& ctx,
  uint8_t* base)`, declared `PPC_EXTERN_FUNC(_xstart)` in `ppc_recomp_shared.h`, defined
  `PPC_WEAK_FUNC` in `ppc_recomp.2.cpp`.
- Every guest-callable function — recompiled game code and kernel import trampolines
  alike — shares one signature: `void(PPCContext& ctx, uint8_t* base)`. Arguments pass
  through `ctx.r3`–`ctx.r10`; return value goes in `ctx.r3`. This means all 210 kernel
  stubs can be generated from one uniform template — no per-function signature research
  needed.
- The full list of 210 required import names was extracted via `grep` over
  `private/ppc/*.h` for `PPC_EXTERN_FUNC(__imp__...)` and saved for stub generation.

## Goal

A new `BigBumpinHost` executable that links `BigBumpinPPC`, sets up the guest memory
image, and calls `_xstart`. All 210 kernel imports get placeholder stubs so the link
succeeds and the game's own code actually executes instead of stopping at a linker
error. The run's outcome (clean return, timeout, or crash) and the log of which stubs
were actually hit becomes the input to Phase 2B.

## Design

### New files

```
host/
├── main.cpp           — memory setup, entry point invocation, watchdog
└── kernel_stubs.cpp    — 210 placeholder kernel import implementations
```

`CMakeLists.txt` gains a `BigBumpinHost` executable target linking `BigBumpinPPC` and
`XenonUtils`.

`kernel_stubs.cpp` is generated once from the extracted import list, then committed as
ordinary hand-maintained source — unlike `private/ppc/`, it is not regenerated by the
recompiler and will be edited function-by-function during Phase 2B as real behavior
replaces placeholders.

### main.cpp responsibilities

1. **Allocate the guest memory image.** `mmap` (or platform equivalent) a
   `PPC_MEMORY_SIZE` (4 GiB) region, read-write, anonymous, private. This is a virtual
   reservation on a 64-bit host — physical pages are only committed where the OS
   actually backs touched addresses, so this is cheap despite the large nominal size.
2. **Load the XEX image into it.** Reuse `Image::ParseImage` (the same XEX loader
   `FindSpecialAddresses` already uses successfully) to parse `private/default.xex`,
   then copy each section's bytes to `base + section.base`.
3. **Populate the indirect-call table.** For every entry in `PPCFuncMappings[]`, write
   `mapping.host` to the address `PPC_LOOKUP_FUNC(base, mapping.guest)` computes.
4. **Initialize `PPCContext`.** Zero it, then set `r1` (stack pointer) to a fixed address
   within the 4 GiB region, in a range that doesn't overlap the loaded image, sized to a
   few MiB. Exact placement is an implementation detail resolved during coding, not a
   design-level decision — any non-overlapping address works since the whole 4 GiB region
   is uniformly writable.
5. **Call `_xstart(ctx, base)`** on a background thread, joined with a timeout (default
   10 seconds, adjustable via a constant). A timeout is a normal, informative outcome —
   report it, don't treat it as a bug to fix in this task.
6. **Report the outcome**: clean return, timeout, or (if the process crashes) whatever
   the OS/debugger naturally surfaces — no custom crash-handling machinery in this phase.

### kernel_stubs.cpp

Each of the 210 stubs follows one pattern:

```cpp
PPC_FUNC(__imp__SomeKernelFunction)
{
    fmt::println("[stub] SomeKernelFunction(r3=0x{:X}, r4=0x{:X}, r5=0x{:X}, r6=0x{:X})",
        ctx.r3.u64, ctx.r4.u64, ctx.r5.u64, ctx.r6.u64);
    ctx.r3.u64 = 0; // benign default; see per-family notes below
}
```

Return-value defaults are chosen per function-name family so the game's own success/
failure checks are more likely to proceed rather than immediately branch into an error
path:
- `Ke*`/`Ex*` status-returning functions (e.g. `KeWaitForSingleObject`,
  `KeReleaseSemaphore`): return a success-shaped value (`0`, matching `STATUS_SUCCESS`/
  `WAIT_OBJECT_0` conventions where the game checks for exactly that).
- `Dbg*` (debug output/breakpoints): no-op, return `0`.
- `ExCreateThread`: returns success but does **not** spawn a real host thread or execute
  the requested guest function — real thread semantics are Phase 2B work. This is a
  deliberate scope cut: it avoids introducing concurrency into Phase 2A entirely, and
  matches the "auto-stub, observe, then fix what's actually hit" strategy — if a test run
  reveals the game depends on a spawned thread's side effects, that surfaces as a 2B
  finding, not a 2A blocker.
- Anything not confidently placeable in a family above: return `0` and rely on the log
  output to flag it for review.

## Success Criteria

Phase 2A is complete when:

1. `kernel_stubs.cpp` defines all 210 imports from the extracted list — link succeeds
   with zero unresolved symbols in `BigBumpinHost`.
2. Running `BigBumpinHost` executes real guest code — confirmed by stub log output
   showing actual kernel calls made during the run (not zero, which would mean `_xstart`
   never really started).
3. The run reaches one of: clean return from `_xstart`, the watchdog timeout, or a crash
   — each is an accepted terminal state, not a failure to fix in this phase.
4. The stub-hit log from the run is saved (not just printed and discarded) so it can seed
   Phase 2B's actual worklist instead of the full undifferentiated 210-item list.

## Explicitly Out of Scope

- Real implementations of any kernel import beyond a benign placeholder (Phase 2B).
- Real thread creation/scheduling for `ExCreateThread` (Phase 2B).
- Any graphics, audio, or input (Phase 3+).
- Handling whatever the game does after a stub returns an unexpected value — Phase 2A's
  job is to observe and log, not to make the game correct.
